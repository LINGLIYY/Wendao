# 修仙・问道 —— 全项目深化方案

> 创建：2026-07-17　更新：2026-07-18　状态：阶段一收尾中

## 一、项目定位

修仙题材文字冒险网页游戏。差异化目标：**游戏骨架由代码保证（数值/判定/存档真实可靠），事件内容最终由 AI 动态生成（区别于市面固定事件表的修仙游戏）**。AI 部分放在阶段三，此前先做纯代码驱动的可玩游戏。

## 二、技术选型决策（已定）

| 决策点 | 结论 | 理由 |
|---|---|---|
| 游戏引擎 | **不用**，继续网页技术 | 文字游戏本质是排版+状态管理，DOM 是最合适的文字渲染引擎；Unity/Godot 为图形/物理设计，纯属负担。同类开源项目（万界道友、仙途）均为网页栈。将来若需 2D 画面，可在网页内嵌 Phaser，无需换方向 |
| 前端 | **Vue 3 + TypeScript + Vite + Pinia** | 中文生态最完善；现有分层可无损迁移：`state.js`→Pinia store、`ui.js`→组件、`data/`→TS 数据模块、`engine.js`→组合式函数。参考项目：仙途（Vue3+TS） |
| 后端 | **Node.js + Hono（TypeScript）** | 前后端同语言；后端核心职责是代理 AI API（密钥不可暴露在前端）与云存档。参考项目：万界道友（Hono）。备选：Python FastAPI（若倾向 Python） |
| 数据库 | **SQLite + Drizzle ORM → 按需升级 PostgreSQL** | 单人游戏 SQLite 零运维、单文件即库；通过 ORM 访问，升级 Postgres 仅改连接配置。多人/云同步之前不上 Postgres |
| 状态存储（阶段一） | localStorage 多槽位（已实现） | 单机阶段不需要任何后端；补充存档导出/导入（JSON 文件）作为防丢手段 |

**总原则：按阶段引入复杂度，每个阶段结束时都是一个完整可玩的游戏。**

## 三、分阶段路线图

### 阶段一：纯前端玩法深化（当前）

目标：把"好不好玩"验证掉。不引入构建工具、后端、数据库。

- [x] 引擎骨架：场景/选项/条件/效果/随机判定
- [x] 多槽位存档系统（手动 6 槽 + 自动档 + 旧档迁移）
- [x] 物品系统深化：物品可使用（丹药回复/功法学习）、装备栏（§6.3）
- [x] 功法系统 v1：功法书研读→习得→主修→被动加成（§6.4）
- [x] 属性面板动态更新与人物面板（§6.1）
- [x] 随机事件池：事件按"类型+稀有度"抽取，为阶段三 AI 生成预留同一接口（§6.2）
- [x] 三栏 CRPG 布局：左人物/中剧情/右背包+事件日志，桌面常显+移动端浮层
- [x] 事件日志系统：场景/选择分类记录，随存档保存（上限 60 条）
- [x] UI/UX 规范化：设计令牌、中文衬线字体、焦点环/Esc/aria-live、动效规范、reduced-motion
- [x] 存档导出/导入（JSON 文件下载/上传）
- [x] 数值体系落地：境界表（突破需求/寿元）、时日计数、灵根显示（§6.5）
- [x] 修炼循环：打坐→灵力增长→突破判定（成功/失败，§6.5）
- [x] 战斗系统 v1：自动回合结算、敌人表、胜负分支（§6.6）
- [x] 内容量：第一章双线可玩循环（青云宗线 / 临江城散修线，各含修炼点+历练区）
- [ ] 寿元与死亡（占位：时日已计数，寿元耗尽判定与战斗死亡 v2 一起做）

### 阶段二：工程化迁移

触发条件：阶段一玩法验证通过、代码量开始难以维护（约 >2000 行）。

- Vite 脚手架 + Vue 3 + TypeScript + Pinia
- 迁移映射：数据层 `data/*.js` → 类型化 TS 模块；`state.js` → Pinia store；`ui.js` → 组件树（正文区/状态栏/物品栏/存档面板）；`engine.js` → 独立纯逻辑模块（不依赖 Vue，方便测试）
- 引入 Vitest 对引擎层做单元测试（判定/效果结算/存档兼容）
- 存档格式带 version 字段（已预留），迁移时写升级函数

### 阶段三：后端 + AI 动态事件（差异化核心）

- Hono 后端：`/api/ai/event`（代理 LLM 生成事件）、`/api/saves`（云存档 CRUD）
- SQLite + Drizzle：saves 表（槽位模型与前端一致）、配置表
- AI 事件生成协议：前端把「事件类型+稀有度+角色状态摘要」发给后端 → LLM 按 JSON Schema 返回结构化事件（文本+选项+效果）→ 引擎照常结算。**AI 只产内容，不碰数值**——数值仍由本地引擎按规则结算，防战力崩坏
- LLM 选型：接口做成 Provider 抽象，支持 DeepSeek/Claude/OpenAI 切换

### 阶段四（可选）：账号与云同步

- 账号系统（邮箱或 OAuth）、多设备同步、SQLite → PostgreSQL
- 触发条件：确实有多设备/多人需求，否则不做

## 五、风险与规避

| 风险 | 规避 |
|---|---|
| 过早工程化，玩法没验证就搭架子 | 严守阶段门槛：阶段一不装任何 npm 包 |
| AI 生成内容破坏数值平衡 | AI 只产文本与"效果引用"，结算永远走本地引擎规则 |
| 存档格式变更导致旧档报废 | 存档 version 字段 + 每次变更写迁移函数（已预留） |
| 内容量不足导致游戏空洞 | 数据与逻辑分离已就位，写剧情=堆 data 条目，可随时众包/AI 辅助产出 |

## 六、核心系统设计详案

### 6.1 属性面板动态更新

**三层属性模型**（单一数据源，显示与判定都从这里取值）：

```
基础值 base（存档里存的）
  + 加成 bonus（装备 stats ＋ 主修功法 passive ＋ 将来的 buff/丹药临时效果）
  = 有效值 eff（面板显示、条件判定、战斗计算统一用它）
```

- 状态栏展示形如 `攻击：8 (+3)`，加成部分单独着色
- 「人物面板」：属性明细（基础＋加成＝有效）、装备槽（可卸下）、功法列表（可切主修）
- 更新机制：阶段一为"状态变更后重渲染"；阶段二迁 Vue 后 bonus/eff 改为 computed 自动响应
- 占位：buff 列表（带持续回合数的临时加成），战斗系统实装时一并做

### 6.2 剧情事件可扩展

**事件 = 场景同格式 + 三个扩展字段**，引擎用同一套渲染/结算逻辑：

| 字段 | 作用 |
|---|---|
| tags | 地点/来源标签（如"临江坊市"），剧情选项以 `eventDraw:{tags:[...]}` 触发抽取 |
| rarity | 稀有度（普通/稀有/传说），按 config 权重加权随机 |
| cond | 出现条件（境界/属性/flag）——低境界抽不到高危事件，呼应基本规则 #7 |

- 扩展方式：往 events.js 加条目＝加事件，零引擎改动
- 占位：一次性事件 once 标记（触发后写 flag 自动排除）、事件链（一个事件解锁后续事件）
- **阶段三 AI 接入点**：AI 按同一 JSON 格式生成事件入池，抽取/结算完全复用——"事件无限可能"的落点在这里

### 6.3 装备道具系统

物品字段按用途叠加：

- 通用：name / type / **rarity 品质**（凡品→灵品→法品→宝品→仙品）/ desc
- 消耗品：use（使用效果，格式同场景 effect，可回复属性/习得功法）+ consumable
- 装备：**slot**（武器/防具/饰品/法宝 四槽）+ **stats**（装备时并入 bonus 层）

装备/卸下在物品栏与人物面板操作，槽位冲突时旧装备自动回背包。
占位：装备强化/炼器、套装效果、灵石购买（见 §七 货币）。

### 6.4 功法系统

功法与"功法书物品"分离：书是道具（use.learnGongfa），功法是独立数据表：

| 字段 | 作用 |
|---|---|
| grade | 品阶：凡/黄/玄/地/天 |
| kind | 类别：心法/攻法/身法…… |
| passive | 主修时的属性被动（并入 bonus 层） |
| cultivateRate | 修炼效率倍率——修炼收益 = 基础值 × cultivateRate × 悟性系数（公式占位，修炼循环实装时定） |

- 可习得多部，主修唯一（第一部自动主修，可随时切换）
- 占位：功法层次（入门/小成/圆满）、神通（战斗主动技，随功法解锁）

### 6.5 修炼循环与突破（数值体系核心）

**境界表升级**：每个境界为对象 `{ name, toNext（突破所需灵力）, lifespan（寿元，先占位） }`。

- **修炼收益** = `(2 + 0~2随机) × 功法效率 × (1 + 悟性×0.08)`，向上取整，最低 1
  - 无主修功法时效率按 0.5（"无功法，事倍功半"——凡人嗑丹药路线仍可行）
  - 每次修炼消耗一日（时日计数，为寿元系统铺路）
- **突破判定**：灵力 ≥ 当前境界 toNext 方可尝试
  - 成功率 = `45% + 悟性×3%`（上限 90%）
  - 成功：境界 +1，消耗所需灵力的 60%，气血 +10 / 攻击 +2 / 防御 +2（肉身蜕变）
  - 失败：灵力损失 30%，气血 -5（走火受创），可再攒再试
- **灵根**：由剧情 flag 决定（天灵根/伪灵根/未测），人物面板显示；后续版本灵根将乘入修炼效率
- 人物面板显示"突破所需：灵力 X / Y"进度，玩家目标始终可见

### 6.6 战斗系统 v1（自动结算）

- **触发**：选项带 `battle: { enemy: 敌人id, win: 胜场景, lose: 败场景 }`
- **敌人表** `data/enemies.js`：`{ name, realm（境界标注，呼应基本规则 #6）, 气血, 攻击, 防御, reward（胜利效果）, rewardText }`
- **结算**：玩家先手，逐回合 `伤害 = max(1, 攻 - 防 ± 2随机)`，30 回合上限（超时判负）；使用**有效属性**（装备/功法加成参战）
- **战败**：气血保留 1 点退回安全点，v1 不死亡（死亡与寿元、复活惩罚在战斗 v2 一起设计）
- **占位**：交互式战斗（玩家每回合选择攻/防/技能/逃）、五行克制、状态效果、神通技能
