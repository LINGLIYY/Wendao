## 一、项目定位

修仙文字冒险 + 回合制成长网页游戏。当前阶段是**纯前端单机游戏**（双击 index.html 即玩）。核心差异化目标在后期（见阶段三）：游戏的骨架由代码保证（数值/判定/存档真实可靠），但事件内容最终由 AI 动态生成——区别于市面固定事件表的修仙游戏。

## 三、技术架构决策

| 决策 | 选择 | 理由 |
|---|---|---|
| 游戏引擎 | 不用 | 文字游戏 DOM 就是最强渲染引擎，Unity/Godot 图形引擎是负担。同类开源项目（万界道友/仙途）全走网页栈 |
| 当前阶段 | 原生 JS，零构建 | 防止"玩法还没验证就搭 Vite/TS 架子"的过度工程化死法 |
| 阶段二 | Vue 3 + TS + Vite + Pinia | 代码量 >2000 行后迁移，现有分层（state.js→Pinia、ui.js→组件、data/→TS 模块）无损映射 |
| 阶段三 | Node.js Hono 后端 + SQLite | 后端唯一使命是代理 LLM API（密钥不能放前端）和云存档 |
| AI 事件生成 | 后端发"事件类型+稀有度+角色状态摘要" → LLM 按 JSON Schema 返回结构化事件 → 本地引擎结算。AI 只产文本，不碰数值 | 从根上防战力崩坏——数值永远由本地规则运行 |

## 四、分阶段路线

| 阶段 | 内容 | 触发条件 |
|---|---|---|
| 一（进行中） | 纯前端单机，核心玩法循环闭环 | 当前 |
| 二 | Vue 3 + Vite + TS 工程化迁移，Vitest 单元测试 | 阶段一玩法验证通过、代码 >2000 行难维护时 |
| 三 | 后端代理 LLM 实现"AI 动态事件"（差异化核心） | 需要对事件内容进行无限扩展时 |
| 四（可选） | 账号系统、云同步、SQLite→PostgreSQL | 有多设备/多人需求时 |

## 五、核心数值公式（供平衡性审查）

- 修炼收益 = `(2 + rand(0~2)) × 功法效率 × (1 + 悟性 × 0.08)`，无功法效率=0.5
- 突破成功率 = `min(0.9, 0.45 + 悟性 × 0.03)`
- 境界突破灵力需求：凡人 10 → 练气 60 → 筑基 200 → 金丹 500 → 元婴 1200 → 化神 3000 → 炼虚 8000 → 合体 20000 → 大乘 50000 → 渡劫 ∞
- 战斗伤害 = `max(1, 攻击方攻击 - 防御方防御 + rand(-2~+2))`，玩家先手，30 回合上限
- 突破成功奖励：气血+10、攻击+2、防御+2
- 初期玩家属性（凡人）：气血 20 / 灵力 0 / 攻击 5 / 防御 5 / 悟性 5 / 勇气 5
